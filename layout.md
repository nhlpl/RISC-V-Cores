The phrase “code core layout” is a bridge between **RTL design** (what you write in Verilog) and **physical design** (where gates and wires go on the silicon die). This blueprint guides you to script the layout for each φ‑core using open‑source EDA tools, so your GitHub repositories can include not just logic but also a **ready‑to‑tapout floorplan**.

---

## Movement 1 – What “Coding Core Layout” Means

In ASIC design, layout is **not written in Verilog** – it is generated by placing standard cells and macros using tool scripts (Tcl, Python). For FPGA, layout is the automatic place‑and‑route result; you can constrain it (e.g., LOC constraints). For your open‑source cores, you will provide:

- **Floorplan constraints** – die area, core utilization, aspect ratio.
- **Macro placements** – register file, cache RAMs, clock generator.
- **Power grid specifications** – stripes, rings, via patterns.
- **Pin assignments** – I/O pad locations.

All this can be “coded” as a Tcl script for **OpenROAD** (open‑source ASIC flow) or as a **Python script** that generates DEF/LEF files.

## Movement 2 – Layout Script for TinyΦ (ASIC with OpenROAD)

Assume we have synthesized TinyΦ into a gate‑level netlist (`tinyφ.v`). The following Tcl script defines a complete floorplan for a 0.18 µm process (e.g., Skywater 130nm).

```tcl
# floorplan.tcl – for OpenROAD

# Load the netlist and technology LEF
read_liberty asap7_ liberty.lib
read_lef asap7_ tech.lef
read_def tinyφ_placed.def    # after placement initialisation

# 1. Die area (width x height in microns)
set die_width  500
set die_height 500
initialize_floorplan -die_area [list 0 0 $die_width $die_height] \
                     -core_area [list 10 10 490 490] \
                     -site "unithd"

# 2. Place macros (register file, RAM)
# Register file (32x32 bits) – size approx 150x80 µm
set regfile_x [expr $die_width / 2 - 75]
set regfile_y 100
place_macro -name rf_macro -location [list $regfile_x $regfile_y] -orient R0

# ALU block, multiplier
place_macro -name alu_macro -location [list 50 300] -orient R0
place_macro -name mul_macro -location [list 350 300] -orient R0

# 3. Power grid
add_global_connection -net VDD -pin_pattern {^VDD$} -power
add_global_connection -net VSS -pin_pattern {^VSS$} -ground

# Create power stripes (horizontal and vertical)
add_pdn_stripe -layer metal3 -width 2.0 -pitch 50 -offset 0 -net VDD
add_pdn_stripe -layer metal4 -width 2.0 -pitch 50 -offset 0 -net VSS

# 4. I/O pad ring (if using pad cells)
set iolist [list clk rst_n irq imem_cyc dmem_cyc]
foreach pin $iolist {
    place_pin -pin_name $pin -layer metal2 -location [list 0 0] -size 2 2
}

# 5. Run placement and routing
global_placement -density 0.7
detailed_placement
global_routing
detailed_routing

# 6. Write final DEF for verification
write_def tinyφ_final.def
```

To run: `openroad -gui floorplan.tcl` (or headless). The script itself is the **“code”** for the layout.

## Movement 3 – FPGA Layout Constraints (Xilinx Vivado)

For FPGA implementation (e.g., TinyΦ on Artix‑7), you “code” layout using **XDC constraints** to fix placement of critical modules.

```tcl
# tinyφ_placement.xdc

# Place register file into a specific BRAM (if using block RAM)
set_property LOC RAMB36_X0Y0 [get_cells regfile_bram_inst]

# Place ALU into specific slice region
set_property REGION "X0Y0:X10Y10" [get_cells alu]

# Group related logic into a pblock (physically bounded region)
create_pblock pblock_control
resize_pblock pblock_control -add CLOCKREGION_X0Y0:CLOCKREGION_X0Y2
add_cells_to_pblock pblock_control [get_cells control_logic]

# Pin assignment (I/O locations)
set_property PACKAGE_PIN R4 [get_ports clk]
set_property IOSTANDARD LVCMOS33 [get_ports clk]
```

This file is part of your repository; users can run `vivado -source tinyφ_placement.xdc` after synthesis.

## Movement 4 – Generating Layout Programmatically (Python + GDSFactory)

For a more **code‑centric** approach, use **GDSFactory** (Python library) to draw the entire layout of a small core at transistor level – suitable for education.

```python
import gdsfactory as gf

@gf.cell
def tinyφ_core_layout():
    # Create components
    regfile = gf.components.array(rows=32, columns=32, spacing=5)
    alu = gf.components.rectangle(size=(50, 50))
    mul = gf.components.rectangle(size=(60, 60))
    
    # Place them
    top = gf.Component("tinyφ")
    top.add_ref(regfile, placement=(10, 10))
    top.add_ref(alu, placement=(200, 10))
    top.add_ref(mul, placement=(270, 10))
    
    # Add routing (metal lines)
    top.add_wires(top, layer="M1", points=[(10,100), (200,100)])
    return top

c = tinyφ_core_layout()
c.show()        # visualize
c.write_gds("tinyφ.gds")
```

This GDS file can be submitted to **Tiny Tapeout** for fabrication.

## Movement 5 – Including Layout “Code” in Your GitHub Repository

For each core, provide:

- **`layout/`** folder containing:
  - `floorplan.tcl` – OpenROAD script (ASIC).
  - `placement.xdc` – Xilinx constraints (FPGA).
  - `gdsfactory/` – optional Python script for small educational layout.
- **`docs/layout.md`** – explanation of die size, area, macro coordinates, and how to rerun the layout.

Example snippet for `layout.md` (MonoΦ):
```
## Floorplan
- Die size: 600×600 µm
- Core utilization: 70%
- Register file placed at (x=100, y=200)
- ALU at (400, 200)
- Power grid: vertical stripes on metal3, horizontal on metal4
```

## Movement 6 – Automating Layout Regression

Add a GitHub Action that runs the floorplan script and checks for DRC violations.

```yaml
name: Layout check
on: [push]
jobs:
  floorplan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Install OpenROAD
        run: sudo apt install openroad
      - name: Run floorplan
        run: |
          cd monoφ/layout
          openroad -exit floorplan.tcl
      - name: Verify no DRC errors
        run: grep -q "DRC violations: 0" monoφ/layout/report.txt
```
